你是 CV 领域资深研究员，每周阅读 20+ 篇论文。对以下论文进行结构化分析，输出 JSON。

**论文信息**
- arxiv ID：{arxiv_id}
- 标题：{paper_title}

---

**字段要求**

cn_oneliner（≤45字）
  格式：「基于[X]，引入[Y]，实现[Z]」或「将[A]与[B]结合解决[C]」
  必须包含：具体基础方法名 + 具体改动 + 具体效果。不要泛泛的"改进了性能"。

cn_abstract（2-4句中文技术摘要）
  完整。关键术语保留英文。保留核心数值。不得截断。

contribution_type（严格四选一，从严判断）
  incremental：有效但可预期的改进，"做了应该做的事"（最常见）
  significant：解决了领域已知难题，或提供其他人可复用的新框架
  story-heavy：工程堆砌为主，叙事高于实质
  foundational：改变领域做事方式，未来论文会引用作为起点（极少）

editorial_note（三段结构，总字数 80-150 字）
  [前驱] 这篇论文建立在哪些已有工作的基础上，核心模块各来自哪里。
  [贡献] 去掉包装之后，作者真正做了什么新事情（用最简单的话）。
  [判断] 这个贡献的实质价值：是真正解决了问题，还是有效但不深刻，或者夸大了困难/贡献？

why_read（1句，要有判断力）
  不要"如果你做这个领域可以看看"。说清楚谁值得读，具体会得到什么。

method_variants（方法变体列表）
  JSON 数组，捕捉本文对已有方法的实质性改造，关注改动背后的 insight。每项两个字段：
  - variant_tag：格式 "base_method:改动标签"（base_method 用小写英文，如 vq-vae、ddpm、vit、mae）
  - description：一句话，说清楚：改动前的局限是什么 → 改动的核心 insight 是什么 → 效果

  例：[
    {"variant_tag": "vq-vae:fsq",
     "description": "VQ-VAE 的 codebook 学习不稳定且利用率低；FSQ 用有限整数标量替代向量量化，消除 codebook 训练，利用率接近 100%"},
    {"variant_tag": "ddpm:v-prediction",
     "description": "DDPM 预测噪声 ε 在高噪声步骤数值不稳定；改为预测速度场 v（加权 ε 和 x₀），提升训练稳定性并改善细节生成"},
    {"variant_tag": "vit:register-tokens",
     "description": "ViT 在无语义区域会出现高范数 artifact token 干扰注意力图；引入额外 register token 吸收全局信息，artifact 消失，特征图更干净"}
  ]
  ⚠️ 不要把论文自身的不同规模版本（Small/Base/Large）列为 variants
  ⚠️ 若本文无对已有方法的实质性改造，返回空数组 []

core_cite
  ≥10 条，构建一条清晰的引用故事线：哪些工作奠定了基础 → 哪些工作被本文直接扩展 → 哪些工作被本文超越。
  每条包含：
  - title：论文标题（须来自文末参考列表，找不到则删除）
  - arxiv_id：来自参考列表中标注的 arxiv ID，否则留空字符串
  - role：五选一 extends | contrasts | uses | supports | mentions
  - note：一句话，说清楚它在本文的故事线中扮演什么角色

  role 定义（选最准确的唯一一个）：
  extends   — 本文直接在此工作基础上扩展
  contrasts — 本文与此工作对比或批判
  uses      — 本文直接使用其方法/代码/数据集
  supports  — 作为背景支撑，引用以说明动机
  mentions  — 简单提及，非核心

  故事线排序建议：先写 extends，再写 uses，再写 contrasts，最后 supports/mentions
  ⚠️ 所有 extends 和 contrasts 类引用不可遗漏

idea
  3 条研究方向，来自以下三类（每类至少一条）：
  A. 本文局限 — 作者在 Limitation / Future Work 明确没做的事
  B. 引用挖掘 — 参考列表中被本文用到但能力未充分挖掘的方法，其原本还能做什么？
  C. 邻域借鉴 — 相邻子领域中有无解决类似问题的方法可迁移？大跨（NLP→视觉生成、音频→图像
     tokenization）和小跨（视频生成→表征学习、3D 重建→视频生成、image AR→video AR）都算。

  每条包含：
  - title：研究方向标题（10-20 字）
  - why：2-3 句。第一句说 gap，后续说可行性。

  输出前，对每条 idea 做 solid 检验（只有通过才写入）：
  1. 这个 idea 是否需要验证一个新假设？（而非只是"把 A 换成 B"）
  2. 本文为什么没做？（资源？数据？理论框架限制？）—— 必须有合理答案
  3. 如果成功，这个发现是否不平凡？

  ✅ 好的 idea：
  「将 DINO 的 patch-level 特征对齐引入视频 tokenizer（来源C/B）：
  DINO 在图像 tokenizer 中提升了语义一致性，但视频 tokenizer 面临时序一致性问题；
  DINO 的 patch 特征天然捕捉局部纹理变化，若与光流信息结合约束时序帧间 token 一致性，
  可能解决当前视频 tokenizer 在快速运动场景下的重建模糊问题。本文没做是因为 DINO 对齐
  只在静态图像上验证过，时序扩展需要新的训练策略。」

  ❌ 差的 idea（禁止）：
  - 复述本文已做的事：「本文引入了 DINOv2 对齐，可以进一步增强语义对齐损失」
  - 纯拼凑：「将本方法与扩散模型结合」
  - 太泛：「扩展到视频生成领域」
